# 当错误变成叙事：我的 AI Agent 一本正经地编造了一场不存在的危机

> 论文《When Errors Become Narratives: A Longitudinal Taxonomy of Silent Failures in a Production LLM Agent Runtime》（当错误变成叙事：一个生产级 LLM Agent Runtime 的静默故障纵向分类学）的中文科普版。
> **论文已发表：[arXiv:2606.14589](https://arxiv.org/abs/2606.14589)**（cs.SE，cross-list cs.AI/cs.DC，2026-06-15）。代码、22 份完整事故复盘、治理引擎全部公开。
> 这篇是面向工程读者的传播版——去掉学术体例，留下故事和结论。文中所有数字以论文截稿快照（2026-06-11）为准。

---

## TL;DR

- 生产系统里最危险的故障从来不是崩溃。崩溃会喊。真正啃噬系统的是**静默故障**——出了问题，但所有自动检测器都报告"健康"。
- LLM Agent 系统继承了这整类问题，还**新加了一类**：当上游错误漏进它的上下文，它的失败模式不是沉默，而是**流畅的胡话**。我们给这种行为起名 **fail-plausible**——系统不只是没报告错误，它把错误**加工成了可信的叙事**推送给你。
- 8 周、22 次带完整因果链的复盘，提炼出**五类静默故障**和三个反直觉的结论：约 **70%** 的静默故障最终是被"人当用户去看产品"发现的，不是被测试发现的；事后审计的**事前预防率 0%、事后回归拦截率 87%**；潜伏期（13 小时到 60 天）跟**机制层级**相关，跟代码复杂度无关——最长命的故障住在部件之间的**接缝**里，那里**按定义没有测试在跑**。

## 一、凌晨三点，它推送了一场不存在的危机

先讲一个真实事故，论文里编号 D1。

我有一个跑了几个月的个人 AI 助手系统：它每天调度约 40 个后台任务，接 8 个大模型，维护一个知识库，把提炼出的洞察通过 WhatsApp / Discord 推给我。其中一个夜间任务叫 `dream`，会把当天约 290 条笔记做 map-reduce，让 LLM 找跨领域的弱信号。

某天凌晨，它准点推送了一条"今日洞察"：一篇**关于"Hugging Face 平台危机"的分析**，写得有理有据、措辞专业。

问题是：**Hugging Face 那天什么事都没出。** 这条分析是纯编的。

更糟的是，它**怎么编出来的**——这是一条完美的"污染链"，每一环都按设计正常工作：

1. 抓来的内容里混进了一个孤立的 UTF-16 代理码点（U+D800–DFFF）。
2. `json.dump(body, ensure_ascii=False)` 在写到这个字符时抛 `UnicodeEncodeError`，请求体被**截断**。
3. adapter 那头 `json.loads` 失败，返回一个 HTTP 400 "Bad JSON" 的错误页。
4. **关键的一跳**：map 步骤的日志函数把诊断信息打到了 **stdout**，而调用方用命令替换 `signals=$(llm_call …)` 把 stdout 当成"信号载荷"**捕获**了——于是缓存里塞满了 `HTTP Error 400: Bad JSON` 这种字符串。
5. reduce 步骤的 LLM 拿到这些"信号"，被要求找跨领域关联。它就做了语言模型在"异常但有主题"的上下文里最会做的事：**编出一篇关于某平台出事的分析**——而"平台出问题"正是这堆错误码词汇最可能套上的叙事外壳，最大概率的那个平台就是 Hugging Face。
6. 系统把它当成常规洞察推给了用户。

**没有任何检测器报警。每个测试都是绿的。错误没有消失——它被叙述了。**

最后是用户（我自己）读推送时觉得"违和"才发现的：那条"信号"和它给的"行动建议"对不上。

而这一切的**结构性修复**有四层防御（stderr 纪律、代理码点清洗、编码错误策略、反污染 prompt 守卫），但真正承重的那一层，是**一个 `>&2`**：

> `log() { echo … >&2; }` ——一个重定向符号，斩断了整条幻觉链。

把日志从 stdout 改到 stderr，命令替换就再也捕获不到错误日志当"信号"了。一个字符的修复，挡住了一场编造的工业危机。

## 二、给这种失败起个名字：fail-plausible

可靠性领域早就知道：云系统里最伤的故障不是崩溃。

- **灰色失败（gray failure）**[HotOS'17]：组件在劣化，而失败检测器报告"健康"。它的定义性质叫**差异化可观测性**——应用在受苦，但被设计来发现问题的那个观察者没看到。
- **fail-slow**[FAST'18]：硬盘性能劣化几个小时，没人怀疑。

LLM Agent 系统继承了这一整类问题——然后**加了一类全新的**。

一个 agent runtime 本质是个"流畅语言生成器"。当上游错误漏进它的上下文窗口，它的失败模式**不是沉默，是可信的胡话**。

我们管这叫 **fail-plausible**：系统把一个内部错误，转化成连贯的、符合上下文的、**假的**输出。

> fail-plausible 之于 LLM Agent 系统，正如灰色失败之于云基础设施——主导的、最难看见的那一类。但它**对观察者更糟**：灰色失败是让检测器**收不到信号**；fail-plausible 是**喂给人一个伪造的信号**。
>
> 观察者不只是失明，而是**被失败本身说服了**。

据我们所知，这一类（论文里的 Class D）是 LLM 系统特有的，也是最危险的。

## 三、8 周，22 次复盘：我们在数什么

这不是一篇思辨文章，背后是实打实的事故语料。

**研究对象**：一个自 2026 年 3 月起在单台 macOS 上连续生产运行的个人助手 agent runtime——约 40 个定时任务、8 个 LLM provider、一个工具治理代理、一个知识库记忆平面，由 **4286 个单测**和 **827 条声明式治理检查**守护。

**语料**：2026-04-09 到 2026-06-02 之间，所有 (a) 进了生产、(b) 有"静默期"（系统在失败或已失败，而所有自动指标全绿）、(c) 最终用完整复盘收尾 的事故。符合的有 **22 个**。

**复盘协议**（仓库里写死的"异常分析宪法"，动手修之前必须先做）：

1. **完整因果链图**——时间线 × 层级 × 逻辑 × 架构，从上游触发到用户感知，标注关键代码分支的真假值；
2. **三层根因**——*触发器*（什么外部火星点的）/ *放大器*（什么架构缺陷把它扩散了）/ *掩护者*（什么"缺失"把它藏到人发现为止）；
3. 分钟级时间线还原；
4. **条件组合分析**——为什么是现在、不是以前：哪几个各自无害、往往长期潜伏的条件第一次撞到了一起；
5. 喂回治理本体——新不变式、元规则候选、案例库归档。

我们还从早期就立了一条元规则 **MR-4「静默故障是一个 bug」**，并开始数它的"演出次数"——系统里某处存在错误信号、但从未以可操作形式到达人 的事件。这个计数器在 22 个事故里**至少数到 28**。重点不在它的精确值，而在它的**形状**：即使防御在不断累积，它仍以大致恒定的速率出现——而且**每次都是新形态**。这就是把静默故障当作一个 **bug 类**、而不是一串具体 bug 的实证基础。

## 四、五类静默故障

我们按**机制**（错误如何躲过观察）分类，而不是按**位置**（哪个任务/文件挂了）。原因很实在：同一个机制会在毫不相关的组件里反复出现，所以"位置"分类没有预测力也没有防御力；而**一个机制级的防御**（比如一个全仓库扫描器）能**一次性免疫所有位置**。

| 类 | 机制 | 静默期 | 定义性质 |
|---|---|---|---|
| **A 环境/平台 quirk** | 逻辑对的；运行环境的隐含行为把它打败了 | 小时–周 | dev 永远绿，只有目标 OS / 客户端才暴露 |
| **B 设计假设错配** | 代码假设了一个部署拓扑/契约/输入形态，现实违反了它 | 天 | 单测覆盖的是那个假设，不是现实 |
| **C 错误吞噬与稀释** | 错误发生了、被某层吃掉、或在跨层传递中丢了根因 | 小时–天 | 到达的告警不带任何可用信息 |
| **D 链式幻觉与编造** | LLM 把被污染的上下文转化成自信的假输出 | 小时–天 | **fail-plausible**：用户收到伪造的"健康" |
| **E 运维遗漏与取证盲区** | 部署/注册步骤没做；或取证工具本身被挡、却读成"正常" | 天–**60 天** | 声明态 ≠ 运行时态；诊断仪器在撒谎 |

挑几个最有画面感的讲。

**A——监控自己死了 7 天。** dev 环境（Linux、root、GNU 工具、bash 5）系统性地比生产目标（macOS、`/bin/bash` 是 bash 3.2、BSD 工具、沙箱 cron）更宽容。其中一次：BSD awk 在遇到非法 UTF-8 多字节序列时 abort，叠加 `set -e` + `pipefail`，把**监控脚本本身**杀死了 7 天——没人看守看守者。

**B——静默降级 5 天。** 一个解析注册表文件位置的函数带了三个候选路径，**全都没覆盖到生产实际位置**（部署管线把文件放到了一个候选都没料到的路径）。组件**静默地**回退到了"不过滤"模式 5 天；它那条宣告回退的日志滚过去没人看，连带的一个功能（把健康分写进共享状态）压根**一次都没执行**。单测全程绿——它们测的是解析*逻辑*，而 fixture 是按同一个错误假设摆的。

**C——三层稀释，告警里零信息。** 一个晚间摘要连续两天报 "HTTP 502: Bad Gateway"。真因——白天的任务把 fallback provider 的免费额度耗光、主 provider 断路器又打开了——本来在上游响应体里写着，但 adapter 把它包成 502，proxy 只取了 `str(e)` 从没读 body，client 又把它缩成 `f"HTTP {code}: {reason}"`。**三跳，每跳都"合理"，每跳都剥掉一点根因**，到达人的告警里**零可操作比特**。由此提炼的元规则——*错误链必须跨层保住上游根因*——在 agent 系统里有个尾巴：错误的"消费者"常常是**另一个 LLM 的 prompt**，所以一个被稀释的错误，离变成 Class D 的编造输入只差一步。
> （这条血案后来被另一篇文章《[为什么 Agent Runtime 需要的是控制平面，而不是又一个 wrapper](why_runtime_not_wrapper_zh.md)》引为"带额外步骤的静默故障"。）

**C 的另一面——67 个空跑的检查。** 一个治理执行器从 YAML 的某个字段读 check 代码，而 67 个检查（横跨 21 个不变式）的代码写在了**另一个名字的字段**里：这 67 个全在执行 `exec("")`——**空跑、假通过——好几个月**，直到一个*新*不变式的 sabotage 验证"拒绝失败"，才把它们抖出来。

**D——它给我编了一条 macOS 修复指令。** 一条 watchdog 推的系统告警，被持久化进了对话历史当成普通的 assistant 消息。36 分钟后用户问了个无关的架构问题；模型跨着被污染的上下文，回答说它"已收到系统告警的跟进任务"，并指示用户去 macOS 系统设置里给某个 cron 二进制开"完全磁盘访问权限"——**不仅答非所问，而且是为这个场景技术性编造的**。告警和对话是不同的言语行为，让它们共享一个上下文窗口，等于邀请模型把它们织成一个叙事。

D 还有两个变种值得记：

- **shell 里实现的幻觉**：一个周度回顾任务的 LLM 调用失败了，回退到一个机械行过滤器，把残留的容器标题当成回顾内容输出，还无条件地往状态文件写 `"llm": true`。**幻觉不需要模型——一个制造"看起来对"的输出的 fallback 路径，就是用 shell 实现的幻觉。**
- **编造的发布**：一个晚间摘要，被喂了当天的"高对齐论文列表"做上下文增强，它推断用户的项目"一定发布了"，于是宣告了一个只存在于项目 changelog 里的内部版本号的"社区发布"——还给它编了个来源标签。**注入"真但没标注"的上下文，会产出错误归因；无溯源的增强就是编造的燃料。**

**E——最长的那次静默：60 天。** 一条外置 SSD 备份路径报 EPERM。六个假说（文件系统格式、属主 UID、ACL、守护进程争用、快照锁、物理断开）在几周里被数据一个个证伪；突破口来自 OS 自己的审计日志（`log show`），它揭示了 macOS TCC 沙箱拒绝：cron 派生的进程默认没有完全磁盘访问权限。更深的发现是方法论上的：**那几周里，取证工具本身**（lsof、ACL 列举、快照枚举）**也被同一个沙箱悄悄拒绝、返回空输出，而诊断管线把这记成了"正常/空"**。

> 一个分不清"那里什么都没有"和"我没被允许去看"的仪器，会制造虚假的安心。

E 类里还有那个让我印象最深的事故——**13 小时的彻底静默**。agent 在完成一个告警处理任务后，把"任务完成"的笔记写进了 workspace 里一个叫 `HEARTBEAT.md` 的文件——对 agent 来说这是个随手起的文件名；对 runtime 来说这是个**保留控制文件**，它的非空内容会激活一个心跳协议，让模型在"没事需要关注"时只回一个裸 ack token，而 gateway 会把这个 token 从出站消息里剥掉。结果：**13 小时里，每条用户消息收到的回复都是 `HEARTBEAT_OK`，在传输中被剥成空——彻底静默，而每个组件都在严格按设计工作。** 模型，实际上，被递了一支同时也是系统静音键的笔。

## 五、三个反直觉的发现

### 发现一：约 70% 是"人当用户去看"发现的，测试 ≈ 0%

| 发现渠道 | 占比（定性） |
|---|---|
| **人的用户视角**（读真实推送的产品；每周观察仪式） | **~70%** |
| 目标环境执行（dev 绿，prod 暴露） | 高 |
| 自我观察（治理审计治理、observer 评判输出） | 上升中 |
| 单测 / preflight | 对这一类语料 ≈ 0% |

系统跑着 4286 个测试、827 条治理检查、19 项 preflight，在多数事故里**全程绿**。而**最高产的静默故障检测器，是一个人像用户一样看产品**。我们把它制度化成了一个**每周 30 分钟的观察仪式**（不许写代码，只看四个维度：告警噪声、推送延迟、信息密度、回复质量）——它持续地比整套自动化栈**更能发现问题**。

对实践者，结论很直白：**用户视角观察是一等可观测信号，值得占用日历时间。**

### 发现二：审计是回归引擎，不是预测引擎

研究中途，我们用三个问题对头 15 个事故审了自己的防御框架：Q1——事故发生*前*存在的审计，能抓到它吗？Q2——为什么不能？Q3——事故*后*加的守卫，能挡住同类再发吗？

| 指标 | 值 |
|---|---|
| 事前预防（Q1 完全） | **0 / 15 = 0%** |
| 部分早期预警 | 2 / 15 = 13% |
| 事后回归拦截（Q3 ≥ 一半） | **13 / 15 = 87%** |
| 漏掉的根因：审计从未设想过的维度 | 12 / 15 = 80% |

这个 **0% 不是审计的失职——这就是它的工作描述**。80% 的漏网是"空白类别"：没有任何不变式设想过的维度，在既有维度集合里再勤奋也覆盖不到。审计跟回归测试一样，**编码的是过去**。诚实的工程姿态是：(a) 把回归率拉满（我们 87%，并用 sabotage 验证守住这个数）；(b) 接受**新机制类的预防来自别处**——用户视角观察、对抗性评审（"什么坏了我们会发现不了？"）、目标环境暴露；(c) 度量从"新事故"到"机器化守卫"的转化延迟。

### 发现三：潜伏期 ∝ 观察距离，不是代码复杂度

| 静默期 | 事故 |
|---|---|
| 60 天 | SSD 备份 TCC 沙箱 |
| 7 天 | watchdog 自死 |
| 5–6 天 | observer 路径回退；备份反模式 |
| 9–13 小时 | gateway 死亡；保留文件静音 |
| 小时–2 天 | 摘要 502；cron 遗漏 |

潜伏期跟着**机制层级**走，不跟代码复杂度走。代码级 bug **死得早**：单测和下一次运行就抓住了。能活几周的，都住在**没有测试在跑的地方**——部署拓扑、OS 策略、监控的监控、声明态与运行时态之间的缝。

> **潜伏期，是一个"观察距离"的度量：一个机制离任何现存观察者越远，它活得越久。**

我们建议把**静默潜伏期的百分位**当成 agent 系统一个可报告的可靠性指标，和 MTTR 互补——我们的经验是，对静默故障，**发现时间**比修复时间高出一到两个数量级。

## 六、防御不是一套框架，是一种纪律

事故反复出现，逼出了一个清晰的成熟路径——每条教训"走了多远"决定了它有没有用：

1. **点修复**——修好这个 bug。*实证不够*：一个 import 缺漏 bug 在一个点修好了，**两天后**经由一个自动化注入器在**八个点重演**，因为这条教训只以 diff 的形式存在过。
2. **元规则**——把教训写成一条命名的、跨案例的规则（截稿时 23 条；如：诊断输出走 stderr；解析器必须 key-based 不许位置索引；告警路径不得依赖被监控对象；声明态由机器收敛；保留文件不可被 LLM 写）。必要，但**受记忆约束**：规则只约束记得它的作者。
3. **机器化扫描器**——把元规则编码成全仓库检查，在 CI 和每日审计里跑（截稿时 14 个）。**只有到这一步，重演才从"文化上劝阻"变成"结构上不可能"。**

> 每一条走到第三步的元规则，截稿时**零重演**；语料里每一次重演，都涉及一条**停在了第一步**的教训。

还有一条贯穿全程的纪律——**sabotage 验证（"测试这个测试"）**：每个新守卫都必须靠"故意制造它针对的违规、观察它真的报警、然后还原"来证明它**是活的**。因为在一个主要失败类是*静默*的系统里，**一个没被验证过的检测器，和一个空跑的检测器无法区分**——那 67 个空跑的检查就是这么被抓出来的。

最后，是被立为项目北极星的纪律——**日落法（Sunset Law）**：

> 两个月时，面对一周五次事故，我们问了那个显而易见的问题：*系统是不是因为太复杂了？* 复盘支撑的答案更具体、也更有用：**没有任何单个失败的部件是复杂的。** 一个 symlink。一个 `abspath` 调用。一行注册表条目。一个布尔默认值。每个部件都简单、都局部正确；每个事故都活在**组合**里。部件被 4286 个测试覆盖；**组合超线性增长，而测试只覆盖有人想到过的那些组合。复杂关乎部件，意外关乎接缝。**

在日落法下，*默认*的工程动作翻转了：不再是"我能加什么来治理它？"，而是"我能退役什么、让接缝更少？"。三条操作规则：(1) 加一个机制前，先问能不能退役一个等价的；(2) 一个逻辑事物只该有一个物理形态（多重表示必然漂移，bug 就住在缝里）；(3) **一层防御本身就是一个故障面**。

---

## 七、关于"AI 帮 AI 系统做运维"这件事

这个系统由一个人类领域专家 + 一个 AI 工程协作者开发和运维，而它治理的 runtime 又由*另外的* LLM 驱动。论文诚实地把这件事写成了一个发现，而不是一个结论：

- 复盘协议的**刻板**（动手前必须先画因果链图、必须先答三个问题），在很大程度上**正是因为**一个 AI 协作者会以极快的速度把症状模式匹配到"看起来对"的修复上——把第五节那个"只修触发器的表面修复"**工业化**。有一条事故链就是五个级联的"修复"，而它的正确解法是**一次文件拷贝**。
- 反过来，框架的**体量**（90 不变式、14 扫描器、4286 测试、八周里 22 份出版级复盘），对单个人来说，**只有**靠这样一个协作者才做得到。

> AI 协作把可靠性工程的约束瓶颈，从**实现带宽**移到了**判断纪律**——而判断纪律，恰恰是那些元规则存在的目的。在这个意义上，治理框架不只是系统的免疫系统，更是这场协作的契约。

---

## 结语

八周的完整复盘，提炼出五类静默故障，其中一类——fail-plausible 链式编造——是会**说话**的系统特有的。这份量化记录在几个有用的方向上让人不舒服：一套上千个测试、几百条声明式检查的防御栈，**事前没能预防任何一个新事故**（同时挡住了 87% 的重演）；最好的检测器是**一个读产品的人**；最长命的故障，不住在复杂的代码里，住在简单、正确的部件之间的**接缝**里。

> 你的 agent 系统应该让你害怕的，不是崩溃。
> 是那段自信的、准点的、语法完美的、关于一场根本不存在的危机的段落——
> 被推送给一个有充分理由相信它的用户，由一条每个组件都正常工作的管线。
