# 为什么 Agent Runtime 需要的是控制平面，而不是又一个 wrapper

> 这是 *[为什么 Agent 系统首先需要一个控制平面](why_control_plane_zh.md)* 的续篇。
> 上一篇论证 agent 系统**需要**控制平面，这一篇论证更难的一点：市面上大多数所谓
> "agent runtime" 其实是 **wrapper**，而两者的区别，是唯一能在生产环境活下来的东西。
> 我用两次相隔八周、互不相识的代码级专家评审作证据，并正面回答最强的质疑——
> *"这不就是个过度工程的 wrapper 吗？"*

---

## TL;DR

- **wrapper 是把请求转发给 LLM API；控制平面是治理这个请求*周围*发生的一切**——provider 路由、降级、断路、工具治理、请求整形、声明态与运行时态的漂移收敛、可观测性。区别从来不在模型，**在接缝（seam）**。
- **两次独立的代码级评审——2026-04-17 与 2026-06-11——都给"控制平面"定位打了约 9/10。而且两次都点中了同一个软肋**：可复用的框架与作者的个人助手深度耦合，复杂度组合面偏大。两个陌生人在同一个优点*和*同一个缺点上撞车，这不是噪声，是信号。
- **控制平面凭什么存在？凭它能被抽离出来。** 对"过度工程的 wrapper"这个质疑，诚实的答案不是否认，而是**日落法**：降复杂度优先于加功能。证据是——一个第三方把治理引擎从 wheel 装进隔离 venv，带着自己的 YAML 跑起来，**零访问本仓库**，照样拿到工具治理 + 语义查询 + governance 审计。

---

## 一、wrapper 陷阱

随便翻一篇"手把手搭 agent runtime"的教程，画的图都长一个样：

```
用户 → [你的代码] → LLM API → 回复
```

那个 `[你的代码]` 框，就是 wrapper。它做请求拼装、可能加点 prompt 模板、可能跑个 tool loop。看着干净，demo 跑得漂亮。但它对那些真正决定"能不能上生产"的问题，**一个都答不上来**：

- **provider 连续返回 503 十分钟。** wrapper 是每个请求都失败？还是无限重试把延迟堆爆？它没有"切到第二个 provider"的概念——因为 wrapper 不持有"有哪些 provider""它们健不健康"这种状态。
- **模型幻觉出一个工具调用**——工具名对，参数 key 错了（`file_path` 写成 `path`）。wrapper 原样转发，工具抛异常，用户看到一坨堆栈。wrapper 修不了，因为它手里没有可以拿来对照修复的 schema。
- **对话涨到 300KB。** wrapper 原样转发，请求超时或者把整个上下文窗口吃光。wrapper 整不了形，因为它没有预算模型。
- **一个后台任务不再触发了。** wrapper 根本不知道这个任务存在。没有"应该在跑的"可以去对比"实际在跑的"。
- **两个记忆层给出矛盾的信息。** wrapper 没有置信度、新鲜度、优先级的概念；它就是拼接然后听天由命。

这些**都不是能力问题**。把模型调得更聪明，一个都解决不了。它们是治理问题，而且全都住在同一个地方：模型与世界之间的**接缝**。wrapper 按定义就是假装这些接缝不存在；控制平面的全部工作，恰恰就是治理这些接缝。

> 上一篇讲的是三平面模型（控制 / 能力 / 记忆）。这一篇收窄到一个更锋利的点：**wrapper 和 runtime 的分界线，就是接缝到底有没有被治理。**

---

## 二、控制平面与 wrapper 的*结构性*区别

控制平面的能力，是 wrapper **做不到**的——不是"懒得做"，是*结构上做不到*，因为这种能力需要"系统怎么运行"的状态，而不只是"转发什么"的状态。

| 关注点 | wrapper | 控制平面 | 本项目里 |
|---|---|---|---|
| **provider 路由** | 写死一个 endpoint | 按能力打分的降级链 + 每 provider 断路器 | `adapter.py` 从 provider 能力声明构建 fallback chain；连续错误后断路器打开，路由到下一个健康 provider |
| **工具治理** | 模型吐什么转发什么 | 白名单（24→12）、按 schema 修参数、硬上限 | `proxy_filters.py` 截断工具集，`fix_tool_args` 修复畸形/幻觉调用，`MAX_TOOLS=12` 强制 |
| **请求整形** | 转发原始 payload | 截断超长历史，保系统/保最近，强制字节预算 | `truncate_messages` 把对话压在硬上限内，最旧先丢 |
| **漂移收敛** | 没有"应该"的模型 | 声明态 ↔ 运行时态，机器检测（可选机器同步） | 收敛框架：5 个 spec，3 个升级到 `machine_sync`——*"声明态必须由机器而非记忆收敛到运行时态"* |
| **自我治理** | 没有 | 不变式 + 元规则 + 扫描器，连治理本身都审计 | 一个治理引擎（截至 V37.9.148：90 不变式 / 23 元规则 / 14 发现扫描器）——包括审计审计器自己的不变式 |

把中间那一列再读一遍。每一项都需要 wrapper 没有的东西：一个关于**系统自身如何运行**的模型。有哪些 provider、健不健康；哪些工具合法、schema 是什么；消息预算多大；什么*应该*在跑、什么*实际*在跑；哪些不变式必须成立。wrapper 不持有这些状态，因为持有它从来就不是 wrapper 的职责。**这些状态，就是控制平面。** 模型在这一切的下游。

这也是为什么"加个 retry 加个 try/except"不能把 wrapper 变成控制平面。retry 是个战术；控制平面是战术被*治理*的地方——"retry"在这里变成"retry 正确的那个 provider、rettry 这么多次、用这个退避，如果整条链耗尽，就把*原始的上游根因*抛出来，而不是一个被稀释成 `HTTP 502` 的玩意"。最后这半句是这里真实的血案：一条错误链在三层传递中丢掉了根因，那就是带额外步骤的静默故障。

## 三、外部验证——以及两位评审都点中的同一个软肋

定位这东西，吹很便宜，验证很贵。所以这里给验证：本项目被两位互不相识的评审、相隔八周、各自在代码层面审过两次。

**评审一（2026-04-17）。** 结论：*"已经是一个雏形完整的 Agent Runtime Control Plane。"* 第一个点名的优点：**控制平面意识成熟。** 主要劣势：*"读起来像一个超强的脚本系统，模块边界偏重作者的心智模型"*，以及*"环境耦合高、可迁移性不足"*。

**评审二（2026-06-11）。** 综合 7.8/10。**定位打了 9/10**——*"从 bridge 到 control plane 的战略跃迁"*。劣势：**个人助手耦合深**，以及**复杂度组合面大**。

这一对评审有两点值得说。

第一，**优点是外部可读的。** 两个陌生人各自读代码，都独立地把"控制平面"当成主导的、正确的定位——不是"LLM wrapper"，不是"聊天机器人"，不是"一堆 cron"。**当一个项目自称的定位，恰好是不相关的专家独立得出的定位，这个定位就是真的。** 这就是营销和架构的区别。

第二——也是不舒服的那部分——**两位评审都点中了同一个缺点。** 评审一叫它"偏重作者的心智模型"，评审二叫它"个人助手耦合深 + 复杂度面"。相隔八周、毫无串通，指向同一道接缝。**当两个独立观察者在一个缺点上收敛，那就不是某一个人的口味问题，是一笔成本。** 假装它不存在，正是一个控制平面悄悄烂回"带额外步骤的 wrapper"的方式。

## 四、诚实的反驳：一个无法被抽离的控制平面，*就是*一个过度工程的 wrapper

所以认真对待最强的那个质疑，因为评审就是这么做的：*"你搭了一个个人助手——约 40 个 cron、一个记忆平面、一个人格文件——然后管这一坨叫控制平面。这不就是个过度工程、外加把你生活绑上去的 wrapper 吗？"*

"否认"式的回答很弱："不是的，你看这么多治理。"——更多治理不是反驳，**更多治理正是这条指控本身。** 一个钉了 90 个不变式的 wrapper 仍然是 wrapper，只是更难删了而已。

真正的回答是一条纪律，是在一周连环事故把代价摊开之后，被明确立为项目北极星的：**日落法**——*降复杂度优先于加功能*。它背后的推理，是这两次评审逼出来的最锋利的一句话：

> 那些事故不是因为系统**复杂**（部件难懂）。是因为系统**组合**——简单的、各自正确的部件，其交互面积超线性增长，超过了测试能覆盖的范围。**复杂关乎部件，意外关乎接缝。**

在日落法下，*默认*的工程动作翻转了：不再是"我能加什么来治理它？"，而是"我能退役什么，让接缝更少？"。三条操作规则——(1) 加一个机制前，先问能不能退役一个等价的；(2) 一个逻辑事物只该有一个物理形态（多重表示必然漂移，bug 就住在缝里）；(3) **一层防御本身就是一个故障面**——"再加一层治理"往往不是答案，"缩小接缝"才是。

但北极星在被兑现之前，也只是又一个声明。下面是兑现，也是 wrapper 永远产不出的那部分：

**治理引擎被抽离成了一个独立、可 pip 安装的包**——`openclaw-ontology-engine`——带一道硬的二层切分。Layer 1 是项目无关的引擎（工具本体、策略评估、五个 check 执行器、收敛框架）。Layer 2 是**你的** YAML，由两个环境变量注入。而"第三方不需要本仓库也能跑"这句话，不是 README 里的一句承诺——它是一道 **CI 回归守卫**：

> 离线 build wheel。装进一个**隔离的** venv，`--no-deps`，没有任何 `PYTHONPATH` 指回仓库。把一个全新的 toy 项目（`LibraryBot`——工具、域、不变式都不同于 bridge，也不同于更早的 WeatherBot demo）**复制出仓库到 `/tmp`**。用*发布的*包名 `import ontology_engine` 和装好的 console scripts 跑引擎。断言它审计的是 LibraryBot——并且断言**不**注入 config 时，同一个 wheel 回退到它*自带的* bundled 默认，证明 config 注入（而不是仓库、不是 `PYTHONPATH`）才是机制。

这道测试在每一次回归里都跑。而且它像这篇文章里每一个论点一样，被防成不是自证——靠 sabotage：把 toy 的上限从 5 改成 7，端到端审计就**失败**，因为它真的在读第三方的源码，不是在复述一个 fixture。

这就是"控制平面而非 wrapper"的可操作定义：**治理与它所治理的应用是可分离的。** wrapper 的逻辑焊死在它那一个 app 上；控制平面的逻辑，是你能拎出来、递给一个陌生人的那一块。

## 五、如果你在搭 agent runtime，带走这六条

1. **产品是接缝，不是模型。** 让 agent 系统活得下来的一切，都在模型与世界之间：路由、修复、整形、漂移、可观测。如果你的 roadmap 是"把 LLM 调用做得更聪明"，你在打磨那个从来不是问题的部分。
2. **控制平面的能力，是 wrapper 结构上不可能有的能力。** 判据：它需要"系统怎么运行"的状态（有哪些 provider、哪些工具、什么预算、什么该在跑），还是只需要"转发什么"的状态？是后者，你还没走出 wrapper。
3. **外部可读性是一个 feature。** 如果两个陌生人各自读你的代码、独立得出同一个词的定位，那是真架构。如果得不出，那是营销——该改的是架构，不是 README。
4. **如果两个陌生人还点中了同一个缺点，那是成本，不是口味。** 收敛的批评是你能拿到的最便宜、信号最高的反馈。花掉它。
5. **可抽离性是"控制平面"与"过度工程 wrapper"的分界线。** 第三方能不能用你的引擎、配自己的 config、零访问你的 app 地跑起来？不能，你的"控制平面"就焊死在一个消费方上——那正是 wrapper 的定义。把可抽离性做成一道*测试*，而不是一句承诺。
6. **先退役，再添加。** 一个只增不减的控制平面，会变回它当初要取代的那个 wrapper，外加假装没变回去的代价。健康 runtime 的默认动作是缩接缝，不是堆防御。"再加一层"是 wrapper 的本能，"删掉一个表示"是控制平面的。

---

## 一句话

wrapper 让模型**可达**，控制平面让系统**可存活**。两者的分界线，是 wrapper 假装不存在的每一道接缝——而你站在正确一侧的证明是：一个从没见过你 app 的人，照样能跑起来你的引擎。
