# openclaw-model-bridge 优化与提升路线图（原因 / 风险 / 对策）

> 目标：把当前“可用的中间件系统”升级为“高可靠、可观测、可扩展、可治理”的个人 PA 基础设施。

## 0. 阅读说明

- 优先级定义：P0（立即做）、P1（近期做）、P2（中期做）。
- 每条都包含：**现状问题 → 具体原因 → 风险 → 对策（可落地）**。

---

## 1) 配置中心化与环境分层（P0）

**现状问题**
- 关键配置散落在多个脚本与环境变量中（provider/model/阈值/端口等），缺少统一 schema 和环境分层（dev/staging/prod）。

**具体原因**
- 项目从单机快速演进，优先满足可用性，配置逐步“就地增长”。

**风险**
- 配置漂移导致线上行为不可预测。
- 变更回滚困难（无法明确“哪一份配置是权威真值”）。

**对策**
- 引入统一配置入口（如 `config/*.yaml` + 环境变量覆盖），并定义 JSON Schema 校验。
- 明确配置优先级：默认值 < 文件配置 < 环境变量 < 启动参数。
- 增加 `config doctor` 脚本：启动前检测缺失项、非法值、冲突项。

---

## 2) Tool Proxy 与 Adapter 的超时/重试策略标准化（P0）

**现状问题**
- 目前存在基础超时处理，但重试策略、幂等边界、错误分类（4xx/5xx/网络异常）未形成统一策略矩阵。

**具体原因**
- 代理层按“先连通再治理”演进，异常处理偏“case-by-case”。

**风险**
- 高峰时段可能出现雪崩式失败（重试风暴或过早失败）。
- 故障定位困难（同类错误在不同路径表现不一致）。

**对策**
- 建立错误分类与重试矩阵（按 endpoint + error type + idempotency）。
- 实施指数退避 + 抖动（jitter），并设置全局重试预算（retry budget）。
- 为请求打上 `request_id`，贯穿 Gateway/Proxy/Adapter/Job 日志。

---

## 3) 端到端可观测性升级（日志、指标、追踪）（P0）

**现状问题**
- 已有 token/error 与健康检查，但缺少统一指标规范与跨组件追踪。

**具体原因**
- 监控以“运维告警可用”为主，尚未系统化到 SRE 级 observability。

**风险**
- MTTR 高（定位慢）。
- 难以做容量评估与成本优化。

**对策**
- 统一结构化日志字段：`timestamp, request_id, session_id, provider, model, latency_ms, tokens, error_code`。
- 暴露 Prometheus 指标（QPS、P95/P99 延迟、错误率、token 消耗、工具调用次数）。
- 引入分布式追踪（至少本地 span 关联），让一次请求全链路可回放。

---

## 4) 测试体系补全：从单测到契约测试（P0）

**现状问题**
- 当前以 `proxy_filters` 单测为主，缺少 Proxy↔Adapter↔Provider 的契约测试与回归样本集。

**具体原因**
- 核心策略层先做了纯函数可测，网络路径测试尚未系统化。

**风险**
- Provider API 细微变化会在生产暴露（schema 漂移、字段变化、SSE 兼容）。
- 重要行为（tool_calls 修复、截断、NO_TOOLS）缺乏端到端回归保障。

**对策**
- 新增 contract tests：模拟 provider 响应样本（成功/限流/超时/坏 JSON/SSE）。
- 引入 golden dataset（真实脱敏请求），每次变更比对输出差异。
- CI 门禁：策略层单测 + 关键路径契约测试必须通过才允许合并。

---

## 5) 安全治理：密钥、最小权限、审计（P0）

**现状问题**
- 已强调环境变量与扫描，但“运行时权限边界”与“敏感操作审计”仍可加强。

**具体原因**
- 当前架构偏单机信任域，默认高权限执行路径较多。

**风险**
- 凭据泄露或误用导致高影响事故。
- 工具链执行范围过大（尤其 `exec`）带来供应链与误操作风险。

**对策**
- 对 `exec` 工具实施命令 allowlist/denylist + 工作目录白名单。
- API key 定期轮转与失效策略（含自动检测即将过期）。
- 审计日志分级：敏感操作（写文件、执行命令、外发消息）强制留痕。

---

## 6) 工具调用治理：策略从“硬编码”向“策略引擎”演进（P1）

**现状问题**
- 工具过滤与参数修复规则主要写在代码中，扩展新工具成本较高。

**具体原因**
- 早期追求稳定，规则内聚在 `proxy_filters.py`。

**风险**
- 新增工具时容易引入回归。
- 多模型差异化策略难以维护。

**对策**
- 将规则数据化：按 provider/model 维度加载策略配置（allow/deny/schema alias）。
- 引入策略版本号与灰度发布（5%→20%→100%）。
- 每次策略变更自动输出“影响分析报告”（影响工具、影响请求比例）。

---

## 7) 请求体与上下文管理优化（P1）

**现状问题**
- 当前主要依赖截断控制体积，对“高价值上下文保留”尚可优化。

**具体原因**
- 先满足 200KB 安全阈值，再做语义层优化。

**风险**
- 关键上下文丢失导致回答质量抖动。
- token 成本上升且效果不稳定。

**对策**
- 引入“语义优先级截断”：system/工具结果/最近多轮按权重保留。
- 增加摘要缓存层（短记忆+长记忆）而非线性堆叠历史。
- 对不同任务类型定义 token budget（问答/检索/执行任务分档）。

---

## 8) 多 Provider 路由与降级策略（P1）

**现状问题**
- 已支持多 provider，但缺少动态路由与自动降级编排。

**具体原因**
- provider registry 解决了“能切换”，但尚未实现“智能切换”。

**风险**
- 单 Provider 波动时整体可用性下降。
- 成本/延迟无法动态优化。

**对策**
- 引入路由策略：按任务类型、时延、错误率、成本进行模型选择。
- 设置 fallback 栈：主模型失败后自动降级（保持工具协议一致）。
- 增加熔断器（circuit breaker）与恢复探测（half-open）。

---

## 9) 定时任务平台化（P1）

**现状问题**
- 任务虽已注册中心化，但执行状态、依赖关系、重跑机制仍偏脚本化。

**具体原因**
- 当前以 cron 驱动，任务编排能力有限。

**风险**
- 任务失败后人工介入成本高。
- 任务互相影响难分析（重叠执行、资源争用）。

**对策**
- 增加任务状态存储（last_run, duration, result, retries, next_run）。
- 实现通用重跑命令：按 `job_id + 时间窗` 回放。
- 加入并发互斥锁与超时熔断，防止任务堆积。

---

## 10) 部署流程从“轮询拉取”向“事件驱动”演进（P1）

**现状问题**
- auto_deploy 采用固定轮询，简单可靠但效率与可观测性有限。

**具体原因**
- 单机场景下轮询成本可接受，优先快速落地。

**风险**
- 发布延迟不可控；错误发布缺少标准化审批与回滚记录。

**对策**
- 增加 GitHub Webhook 触发部署（保留轮询作为兜底）。
- 引入“预发布检查清单”：测试、配置校验、安全扫描必须通过。
- 发布记录落盘（版本号、提交号、执行人、耗时、结果）。

---

## 11) 文档与知识库工程化（P2）

**现状问题**
- 文档较丰富，但存在分散与重复，缺少统一“决策记录”机制。

**具体原因**
- 迭代快，文档按任务增量写入。

**风险**
- 新成员（或未来你自己）理解成本高。
- 架构演进理由不可追溯，重复踩坑。

**对策**
- 建立 ADR（Architecture Decision Record）目录，记录每次关键架构决策。
- 维护“系统现状图 + 依赖图 + 故障树”三件套。
- 文档纳入 CI：链接检查、示例命令可执行性检查。

---

## 12) 成本治理（Token / API / 任务成本）（P2）

**现状问题**
- 已有 token 告警，但缺少按任务、按模型、按时间窗口的成本归因。

**具体原因**
- 当前监控重点在稳定性，成本视角尚未产品化。

**风险**
- 长期运行成本不可控。
- 难判断“质量提升是否值得成本增加”。

**对策**
- 构建成本仪表盘：`cost per task`, `cost per successful outcome`, `cost trend`。
- 对高频任务引入缓存与去重策略。
- 周期性执行“低 ROI 任务”清理。

---

## 13) 个人 PA 目标导向的能力升级（P2）

**现状问题**
- 当前侧重桥接与运维，尚未形成“个人 PA 的任务闭环能力模型”。

**具体原因**
- 基础设施先行，产品能力层尚待系统化设计。

**风险**
- 系统长期停留在“能跑”，难上升到“能持续创造价值”。

**对策**
- 以用户价值定义能力域：信息情报、决策辅助、执行自动化、复盘学习。
- 给每个能力域定义 KPI（时效、准确率、节省时间、可解释性）。
- 对核心任务建立“人类在环”审批节点（高风险动作必须确认）。

---

## 建议的推进顺序（90 天）

- **第 1-30 天（稳态优先）**：1/2/3/4/5（配置、可靠性、观测、测试、安全）
- **第 31-60 天（扩展优先）**：6/7/8/9（策略引擎、上下文优化、多模型路由、任务平台化）
- **第 61-90 天（价值优先）**：10/11/12/13（发布体系、知识工程、成本治理、PA 能力闭环）

> 一句话：先把“系统不掉链子”做到极致，再做“系统更聪明”，最后做“系统更有长期复利”。
